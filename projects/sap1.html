Overview

Designed and implemented an 8-bit SAP-1 (Simple-As-Possible) educational processor using Verilog HDL to explore computer architecture from the hardware level. The processor implements the complete fetch-decode-execute cycle through modular RTL components including registers, memory, an arithmetic logic unit (ALU), and a control unit. The project was developed and verified using Xilinx Vivado simulations, providing practical experience in digital design, processor architecture, and hardware verification.

Why I Built This

While learning digital logic, I realized that understanding individual components such as logic gates, multiplexers, and registers only provided a fragmented view of how a processor actually works. I wanted to bridge that gap by building a complete system where every module interacted to execute real instructions.

The SAP-1 computer was the perfect opportunity to move beyond theory and implement a processor entirely in Verilog. Instead of treating computer architecture as a collection of textbook diagrams, I designed and integrated the fundamental hardware blocks responsible for instruction execution, data movement, arithmetic operations, and memory access. This project allowed me to experience how a processor operates at the RTL (Register Transfer Level), where timing, control signals, and module interactions become as important as the logic itself.

One of the most rewarding aspects of the project was watching the processor execute instructions through simulation. Seeing registers transfer data, the control unit sequence operations, and the ALU produce results made concepts that once existed only on paper become tangible. The experience fundamentally changed how I approached computer architecture and gave me a much deeper appreciation for the engineering behind modern processors.

Beyond completing the project, building SAP-1 established a strong foundation for studying Microprocessors and Microcontrollers, allowing me to approach later coursework with practical implementation experience instead of relying solely on theoretical understanding.

Component Details

Clock Generator

Purpose

Generates the synchronous master clock pulses that drive register transitions across all modules. Handles program execution halts dynamically.

How It Works

Routes the primary clock source clk_in to the system output line clk_out. When the controller asserts the active-high halt signal hlt (triggered by a software HLT instruction), the output clock line is clamped to 1'b0, instantly stopping all synchronous modules.

Inputs & Outputs

  • hlt (Input): Halt command from the control unit
  • clk_in (Input): Primary physical oscillator input
  • clk_out (Output): Gated clock pulse distributed to all modules

RTL Logic (Verilog)

module clock(
    input hlt, // halt signal
    input clk_in,

    output clk_out
);

    assign clk_out = hlt ? 1'b0 : clk_in;

endmodule

Design Notes

Rather than using complex state logic to stop instruction execution, the clock gating design uses a single multiplexer assignment. This hardware-level halt effectively stops execution immediately without requiring registers to support custom enable bits.

Program Counter (PC)

Purpose

Tracks the memory address of the next instruction to fetch and execute in the sequence of program execution.

How It Works

Maintains a 4-bit address value. On the rising edge of the clock, if the increment control signal inc is asserted, the count increments by one. The count asynchronously clears to 4'b0 when the active-high reset signal rst is asserted.

Inputs & Outputs

  • clk (Input): Gated master clock
  • rst (Input): Asynchronous reset
  • inc (Input): Active-high increment signal from Controller
  • out (Output) [7:0]: 8-bit bus output containing the 4-bit PC count value

RTL Logic (Verilog)

module pc(
    input clk,
    input rst,
    input inc,

    output[7:0] out
);
	
    reg[3:0] pc;
	
    always @ (posedge clk, posedge rst)
    begin
        if (rst)
        begin
            pc <= 4'b0;
        end else if (inc)
        begin
            pc <= pc + 1;
        end
    end

    assign out = pc;

endmodule

Design Notes

Since the SAP-1 memory model is 16 bytes, the physical program counter is 4 bits wide. However, the output line is assigned to an 8-bit width to match the shared W-Bus width, relying on Verilog's implicit zero-extension during assignment.

Memory Address Register (MAR)

Purpose

Caches target memory addresses during execution phases to ensure memory read addresses remain stable.

How It Works

Operates inside the unified memory module. On the rising clock edge, if the control signal load (MAR Load) is active, the register latches the lower 4 bits of the W-Bus (bus[3:0]). The latched value is used to address the RAM array.

Inputs & Outputs

  • clk (Input): System clock
  • rst (Input): Asynchronous reset
  • load (Input): Active-high address load signal (driven by ctrl_word[8])
  • bus (Input) [7:0]: Data bus containing target addresses

RTL Logic (Verilog)

// MAR Latch Block inside memory module
reg[3:0] mar;

always @ (posedge clk, posedge rst)
begin
    if (rst)
    begin
        mar <= 4'b0;
    end else if (load)
    begin
        mar <= bus[3:0];
    end
end

Design Notes

The MAR is critical because it decouples RAM input addresses from active fluctuations on the shared W-Bus. It takes the computer 2 cycles to read from memory: the 1st cycle loads the address into MAR, and the 2nd cycle uses that address to retrieve RAM contents.

RAM (Random Access Memory)

Purpose

Provides read-only storage containing program code instructions and memory variables.

How It Works

Declares a 16x8-bit memory block. At startup, it reads program bytes from the external binary file program.bin. It reads memory asynchronously, indexing the array with the latched 4-bit MAR address and driving its data lines onto the W-Bus during memory enable states.

Inputs & Outputs

  • mar (Internal Reference) [3:0]: Active lookup address index
  • out (Output) [7:0]: Memory contents at MAR index

RTL Logic (Verilog)

// Memory array definition and asynchronous lookup
reg[7:0] ram[0:15];		// 16 8-bit wide elements

initial begin
    $readmemh("program.bin", ram);
end

assign out = ram[mar];

Design Notes

Memory initialization utilizes Verilog's standard compiler task $readmemh. This loads binary instruction code directly into the simulated RAM array during compile initialization, simplifying simulation test sequences.

Instruction Register (IR)

Purpose

Holds the current 8-bit instruction fetched from memory during execution cycles.

How It Works

On the rising edge of the clock, if the control signal load is active, the register stores the 8-bit data byte from the W-Bus. The stored byte is used to generate outputs for instruction decoding.

Inputs & Outputs

  • clk (Input): System clock
  • rst (Input): Asynchronous reset
  • load (Input): Load instruction control (driven by ctrl_word[6])
  • bus (Input) [7:0]: Shared system data bus
  • out (Output) [7:0]: Latched instruction byte

RTL Logic (Verilog)

module ir(
    input clk, 
    input rst,
    input load,
    input[7:0] bus,

    output[7:0] out
);

reg[7:0] ir;

always @ (posedge clk, posedge rst)
begin
    if (rst)
    begin
        ir <= 8'b0;
    end else if (load)
    begin
        ir <= bus;
    end
end

assign out = ir;

endmodule

Design Notes

In the top module, the IR output is split: the high nibble (ir_out[7:4]) routes directly to the controller module to determine active opcodes, while the low nibble (ir_out[3:0]) contains operand memory addresses to put back onto the W-Bus.

Controller / Sequencer

Purpose

Decodes instructions and sets the control lines to direct the flow of data between all other registers.

How It Works

Includes a 3-bit stage register incrementing from 0 to 5 on every clock pulse. At each stage, it sets a 12-bit control word (ctrl_word) based on the instruction opcode. Stages 0-2 handle Fetch, while stages 3-5 route control signals for execution.

Inputs & Outputs

  • clk (Input): System clock
  • rst (Input): Asynchronous reset
  • opcode (Input) [3:0]: Upper nibble from the IR
  • out (Output) [11:0]: 12-bit active control word
  • stage (Output) [2:0]: Active cycle stage pointer

RTL Logic (Verilog)

// Stage Counter segment in controller module
always @(posedge clk or posedge rst) begin
    if (rst) begin
        stage <= 0;
    end else begin
        if (stage == 5) begin
            stage <= 0;
        end else begin
            stage <= stage + 1;
        end
    end
end

Design Notes

The control word mapping matches these signal values: ctrl_word[11] (SIG_HLT), [10] (SIG_PC_INC), [9] (SIG_PC_EN), [8] (SIG_MEM_LOAD), [7] (SIG_MEM_EN), [6] (SIG_IR_LOAD), [5] (SIG_IR_EN), [4] (SIG_A_LOAD), [3] (SIG_A_EN), [2] (SIG_B_LOAD), [1] (SIG_ADDER_SUB), and [0] (SIG_ADDER_EN).

Register A (Accumulator)

Purpose

Primary accumulator register that acts as the source and destination for ALU operations.

How It Works

On the rising edge of the clock, if the control signal load is active, the register latches the 8-bit value from the W-Bus. The output is continuously routed to the ALU inputs and can also drive the W-Bus.

Inputs & Outputs

  • clk (Input): System clock
  • rst (Input): Asynchronous reset
  • load (Input): Load control signal (driven by ctrl_word[4])
  • bus (Input) [7:0]: Input data bus
  • out (Output) [7:0]: Stored register contents

RTL Logic (Verilog)

module reg_a(
    input clk,
    input rst,
    input load,
    input[7:0] bus,

    output[7:0] out
);

reg[7:0] reg_a;

always @ (posedge clk, posedge rst)
begin
    if (rst)
    begin
        reg_a <= 8'b0;
    end else if (load)
    begin
        reg_a <= bus;
    end
end

assign out = reg_a;

endmodule

Design Notes

Register A acts as the target for all arithmetic execution write-back phases. In Stage 5 of ADD and SUB instructions, the ALU outputs are loaded back into Register A by asserting the load signal.

Register B

Purpose

Helper register storing the second operand during addition and subtraction instructions.

How It Works

Identical to Register A in latching behavior. Loads data from the W-Bus on the rising clock edge when the control signal load is active. Its output is continuously routed to the ALU inputs.

Inputs & Outputs

  • clk (Input): System clock
  • rst (Input): Asynchronous reset
  • load (Input): Load control signal (driven by ctrl_word[2])
  • bus (Input) [7:0]: Input data bus
  • out (Output) [7:0]: Stored operand contents

RTL Logic (Verilog)

module reg_b(
    input clk,
    input rst,
    input load,
    input[7:0] bus,

    output[7:0] out
);

reg[7:0] reg_b;

always @ (posedge clk, posedge rst)
begin
    if (rst)
    begin
        reg_b <= 8'b0;
    end else if (load)
    begin
        reg_b <= bus;
    end
end

assign out = reg_b;

endmodule

Design Notes

Unlike Register A, Register B does not connect to output drivers on the shared W-Bus in this top module design. Its outputs are routed directly to the ALU adder port, reducing bus arbitration logic.

Arithmetic Logic Unit (ALU)

Purpose

Performs core addition and subtraction operations on operand data inputs.

How It Works

Implements combinational arithmetic. It continuously computes a - b when the control line sub is active (high), and a + b when sub is inactive (low).

Inputs & Outputs

  • a (Input) [7:0]: Left side operand (from Register A)
  • b (Input) [7:0]: Right side operand (from Register B)
  • sub (Input): Add/Subtract mode select line (driven by ctrl_word[1])
  • out (Output) [7:0]: Computed arithmetic output

RTL Logic (Verilog)

module adder(
    input[7:0] a,
    input[7:0] b,
    input sub,

    output[7:0] out
);
	
    assign out = sub ? (a - b): (a + b);

endmodule

Design Notes

The ALU is completely asynchronous and does not require clock inputs. It computes results continuously based on changes at registers A and B. Its output is latched back into Register A during execution stage 5 when the control word asserts SIG_ADDER_EN.

Output Register

Purpose

Latches and displays calculated execution values for output monitoring.

How It Works

In this specific implementation, separate Output Register modules are simplified. The final program output state is read directly from the Register A (Accumulator) line in the top-level wire mapping.

Inputs & Outputs

  • reg_a_out (Simulation Probe) [7:0]: Accumulator register output data

RTL Logic (Verilog)

// Top-level testbench monitoring signals
uut (
    .CLK(CLK),
    .rst(rst)
);

assign reg_a_out = uut.reg_a_out;

Design Notes

To reduce overhead, Register A serves as the primary observation point. The testbench monitors reg_a_out to verify calculations (e.g. 3 + 4 - 2 = 5) before execution halts.

Instruction Set

The processor supports a minimal but complete instruction set including LDA (Load), ADD, SUB, OUT (Output), and HLT (Halt). Each instruction is 8 bits wide — 4-bit opcode and 4-bit operand — allowing up to 16 memory addresses.

Key Takeaways

This project deepened understanding of how hardware executes software at the lowest level — from clock cycles driving state machines to how control signals orchestrate data flow across registers and the ALU.
