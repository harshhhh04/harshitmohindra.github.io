Driver Drowsiness Detection

Python • OpenCV • Arduino • Computer Vision

Overview

Developed an engineering prototype to detect driver fatigue and execute safety interventions. The system consists of a Python-based computer vision application and an Arduino-based embedded safety controller. The vision pipeline monitors driver eye states in real time, transmitting detection results over Serial to the Arduino. If eyes remain closed for more than two seconds, the controller activates an audible buzzer and cuts power to a DC motor representing a simulated vehicle drivetrain.

This prototype demonstrates a complete perception → decision → actuation pipeline using accessible hardware. The long-term engineering vision is that, instead of controlling a DC motor, the controller could communicate with an automotive ECU over the CAN bus to request an appropriate safety intervention. This future deployment architecture is a long-term goal, while the current physical actuation uses a simulated motor setup.

System Pipeline & Technical Details

The prototype functions through a serialized data flow across hardware and software components:

  • Webcam Input: Captures real-time video frames of the driver.
  • Computer Vision: Processes frames in Python using OpenCV Haar Cascade classifiers, performing face detection followed by localized eye detection within the face region. Eye states are classified as Open or Closed.
  • Temporal Smoothing: Maintains a sliding window of the last 15 frame classifications. The driver is flagged as drowsy only if at least 60% (9 of 15) of these frames indicate closed eyes, preventing normal blinks from triggering the safety system.
  • Serial Communication: Transmits binary eye state classifications from Python to the microcontroller.
  • Embedded Controller: An Arduino tracks continuous eye closure duration. If closure exceeds two seconds, it activates the buzzer and disables the motor driver.
  • Safety Actuation: Silences the buzzer and restores power to the DC motor immediately upon detecting that the eyes have reopened. Utilizes an L298N motor driver and DC motor setup to represent drivetrain interventions.

Engineering Decisions

Temporal Smoothing vs. Frame-by-Frame Reaction: Implementing a 15-frame sliding window reduces false positives caused by natural blinking or transient face/eye tracking failures. This ensures safety interventions are only triggered by persistent fatigue rather than high-frequency noise.

Independent Controller Threshold Validation: The embedded controller independently runs the safety timer and verifies the two-second threshold rather than relying on the computer vision script. This design pattern separates safety-critical timing and actuation logic from the non-deterministic operating system environment of the host PC. If the Python script crashes or suffers latency spikes, the microcontroller maintains deterministic timing and safety control.

Limitations & Future Improvements

Prototype Limitations:

  • Haar Cascade classifiers are highly sensitive to driver viewing angles and changes in ambient lighting.
  • The alarm implementation utilizes blocking delays in the embedded code, which restricts concurrent controller tasks.
  • The serial COM port configuration is hardcoded within the Python source code.
  • The prototype uses a DC motor and L298N driver to simulate vehicle safety actuation rather than actual automotive interfaces.

Future Work:

  • Replace Haar Cascades with MediaPipe Face Mesh or Eye Aspect Ratio (EAR) tracking to improve feature localization.
  • Refactor the buzzer control logic into a non-blocking state machine using millis() timing.
  • Enhance detection reliability in low-light environments by integrating infrared illumination.
  • Transition from direct motor control to CAN bus integration to support testing with actual automotive ECUs.
  • Incorporate redundant sensors, such as steering angle sensors, to improve system reliability.
